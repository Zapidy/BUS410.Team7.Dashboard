## Reconciliation plan

Five sources audited; here is the verdict and the action item per source.

### 1. CRA disclosure — packaged differently, same content ✓

**Apparent drift:** 2009–2013 ship as a single bundled `exp_discl.dat` (~270 MB unpacked); 2014–2024 ship as 8 separate `cra{YYYY}_Discl_D{11,12,21,22,3,4,5,6}.dat` files.

**Reality:** the bundled file contains all 8 D-record types interleaved. Verified empirically — 2009's `exp_discl.dat` has 2.7M records with the full {D1-1, D1-2, D2-1, D2-2, D3-0, D4-0, D5-0, D6-0} type set, exactly matching 2024 (which has 4.4M records across the 8 split files).

**Action: NONE.** The parser at [etl/cra/parse_cra.py](../etl/cra/parse_cra.py) already filters by record-type prefix (`line.startswith("D6-0")` etc.) regardless of filename. Already correct. Empirically confirmed by the parser successfully extracting 1,234,045 tract-year rows across the 16-year panel.

### 2. CRA aggregate — drift not yet audited ⏳

Aggregate files (the A-records) were not parsed in detail by this audit. They aren't currently used by the panel build, but if we add them as a sanity-check source we should re-audit. **Action: defer; only re-check if/when we use the aggregate files.**

### 3. CRA transmittal — used for stable-reporter cohort ✓

Transmittal files have a single record per reporter per year. Field positions (per FFIEC spec) are stable since the 1996 introduction:

| 0–9 | 10 | 11–14 | 15–44 | 45–84 | 85–109 | 110–111 | 112–121 |
|---|---|---|---|---|---|---|---|
| respondent_id | agency_code | activity_year | name | street | city | state | zip |

The Round-5 parser already uses these positions to extract the lender_id (agency × respondent) for the stable-reporter cohort — already producing **295 stable reporters** of 1,629 ever-seen.

**Action: NONE.** Reconciled at parse time.

### 4. FDIC SoD — fully stable ✓

10 columns identical across all 16 years (2009–2024). The only signal that varies is the **branch count itself** (99,531 → 76,712), which is the desired secular trend, not schema drift.

**Action: NONE.** No reconciliation required.

### 5. SBA loan-level — TWO schema families that need normalization ⚠️

The SBA publishes **two distinct programs** with **different fields**:

| Family | Files | Cols | Lender field | Notes |
|---|---|---|---|---|
| 504 | `foia-504-*.csv` (2 files) | 40 | `cdc_name`, `cdc_street`, `cdc_city`, `cdc_state`, `cdc_zip`, `thirdpartydollars`, `thirdpartylender_name` | Certified Development Companies + a third-party lender |
| 7(a) | `foia-7a-*.csv` (4 files) | 43 | `bankname`, `bankfdicnumber`, `bankncuanumber`, `bankcity`, `bankstate`, `bankstreet`, `bankzip` | Direct bank/credit-union lending |

The 504 program has **CDC + third-party-lender** structure (two parties per loan); 7(a) is **single-bank**.

**Action: at parse time, project both into a unified `lender_name` column.** Already partially done in [parse_sba.py](../etl/sba/parse_sba.py) which uses `bankname or cdc_name` for the lender id. Worth tightening:

- A `program` column (`504` vs `7a`) — already implicit in our aggregator's `n_504`/`n_7a` counts
- A `lender_type` column (`bank` vs `cdc`)
- For 504s, also capture `thirdpartylender_name` as a secondary lender ID (a single 504 loan involves both a CDC and a bank)

### 6. HMDA tract-aggregates — fully stable ✓ (we controlled the schema)

All 7 years (2018–2024) ship 18 columns identical, because we control the aggregation schema in [pull_hmda.py](../etl/hmda/pull_hmda.py). Pre-2018 HMDA is genuinely unavailable through the API.

**Action: NONE.** The `has_hmda=0` flag in the panel handles the 2009–2017 gap.

### 7. ACS 5-year — variable IDs changed in 2014 ✓ already reconciled

Pre-2014 vintages (2010, 2011, 2012, 2013) use legacy variables:
- `B15002_*` for educational attainment (gendered universe + bachelor's male/female + master's male/female)
- `B23001_001E` for employment universe (gross labor-force coverage)

2014+ vintages (2015, 2020, 2022) use modern variables:
- `B15003_*` for educational attainment (single universe, ungendered breakouts at higher resolution)
- `B23025_*` for labor force / unemployed (separates "in labor force" from "civilian labor force")

**Action: ALREADY DONE.** Two separate pullers ([pull_acs.py](../etl/acs/pull_acs.py) for 2014+, [pull_acs_early.py](../etl/acs/pull_acs_early.py) for 2010–2013). The pivot at [pivot_acs.py](../etl/acs/pivot_acs.py) maps both into the same canonical output columns:

- `pct_bachelor_plus` (combines B15002_017+033+034+035 in early years; B15003_022+023+024+025 in modern years)
- `unemployment_rate` (B23001 ratio in early; B23025_005 / B23025_002 in modern)

The two definitions are slightly different (B23025 separates "in labor force" more cleanly than B23001), so the panel includes an `acs_vintage_used` column. For modeling, consider an `acs_schema_v2` interaction flag if any feature behaves discontinuously at vintage 2015.

### 8. Tract-vintage harmonization — separate work item

Census tracts re-draw at each decennial. CRA pre-2012 likely uses 2000-vintage; 2012-2021 uses 2010-vintage; 2022+ uses 2020-vintage. We have BOTH crosswalks pulled (`tract_xwalk_2000_2010.txt` and `tract_xwalk_2010_2020.txt`) but the panel build does **not yet apply them**.

**Status:** the CRA parser preserves whatever tract code is in each year's source file. A "tract 36061003500 in 2008" is not the same polygon as in 2018, so cross-year tract-trajectory features have a known inaccuracy on tract-vintage transitions (~5% of tracts split or merge between vintages).

**Action item for Phase 2.5:** add a `harmonize_to_2020.py` that crosswalks all earlier-vintage tract codes to their 2020 equivalents using population-weighted apportionment from the relationship files. Round-5 model results in [diagnostics/walk_forward/](../diagnostics/walk_forward/) are based on the un-harmonized panel — meaningful improvement is plausible but not yet measured.
